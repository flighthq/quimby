# Quimby — Key Design Decisions

The rationale log for quimby's architecture — why each choice was made, and what was rejected. See [design.md](./design.md) for the concepts and behaviors these decisions shape, and [cli-surface.md](./cli-surface.md) for the command reference.

- **Quimby is a courier, not a post office**: It hand-carries parcels between agents and across the boundary; it does not run a mailroom. There is no standing archive of past work — a handoff is assembled, carried, and dropped. This deletes a whole class of maintenance overhead: no sequence counter, no orphaned artifacts outliving removed agents, no unbounded store to curate. Durable history is git's job.
- **A handoff is one shape, carrying whichever halves exist**: Always a folder — note and/or diff and/or files, with a `meta.yaml` written last as the completion signal. "Pack vs instruction" is not a type distinction, just different contents, so there is one object and one set of verbs to learn.
- **Content-derived names, time in the manifest**: A parcel is `<from>-<contentHash>` — no counter, dedupes identical carries, self-describing. `createdAt` lives in `meta.yaml`, not the name, so identical re-sends stay idempotent instead of piling up; chronology comes from the manifest and the `out/sent/` ledger.
- **The mailbox is an explicit-lifecycle `handoff/` tree — state is a directory level, never a dot-dir**: the mailbox is `handoff/out/{draft,queued,sent}/<recipient>` and `handoff/in/{received,processed}/<sender>-<hash>`, with the live `status/<peer>.md` mirror kept at its own root outside `handoff/`. Making the **state a directory level above the party name** (rather than the earlier dot-prefixed `outbox/.sent`, `inbox/.done`) buys three things: it is self-documenting (the lifecycle reads off the path), it is **collision-safe** (an agent may be named anything — even `queued` or `status` — because the scanner enumerates a fixed known set of state dirs and reads party names only as the leaves, so nothing new is reserved beyond today's `host`/name-validation rules), and it gives one clean glob per tray for "all outgoing" / "all incoming." Direction (`in`/`out`) is grouping, not disambiguation (the states are already directional), kept for the two-tray mental model. `status/` stays **out** of `handoff/` because it is a continuously-overwritten mirror, not a discrete parcel — separating it sharpens the "status is not a parcel" principle.
- **Addressed out tray, content-named in tray**: The two trays answer different questions — "who is this for" (`out/queued/<recipient>`) when authoring, "what did I get and from whom" (`in/received/<sender>-<hash>`) when receiving — so they name parcels differently on purpose.
- **Author-then-publish with an atomic rename closes the partial-write race by construction**: an agent authors a parcel under the **unscanned** `out/draft/<recipient>/`, then publishes it with a single same-filesystem `mv` into `out/queued/`. A directory rename is atomic, so a partial parcel **never appears in `queued/`** — the auto-dispatch double-dispatch / half-written-parcel race is prevented structurally, not by timing. The settle-debounce on `queued/` is deliberately **kept as a cheap fallback** (it protects an agent that writes straight into `queued/`, skipping draft), not ripped out — the layering is: staging prevents the race, debounce is the safety net. This supersedes an earlier `.ready`/completion-marker idea (which needed agent cooperation and a reserved filename); draft-staging needs neither. An atomically-published parcel settles in one cycle, so net latency stays ≤1 poll cycle — no "instant dispatch" claim is made.
- **Non-destructive delivery**: Carry drains a queued parcel only on success, and to an `out/sent/` receipt rather than a delete; a bad address bounces and stays queued. An agent never has to rewrite a parcel because a delivery failed.
- **A verb per movement**: `handoff` moves sideways (agent → agent, or host → agent), `dispatch` enacts an agent's outbox queue, `merge` moves out across the boundary, `assign` sets an agent's task. Handoffs land in the inbox and never clobber `assignment.md`. `handoff` and `dispatch` are separate verbs because the outbox is a distinct mechanism, and because keeping the host the only non-agent source avoids unfulfillable path sources.
- **Directed handoff vs ambient status**: Directed work uses `handoff` (addressed, validated); "to whom it may concern" is `status`, which the server mirrors to every agent and each agent reads on demand (`./agent.sh peers`, then a named peer peek). Broadcast is deliberately not a handoff mode — it would copy into every inbox and make every agent filter, the token cost we are avoiding; ambient status stays a pull, so it never lands in an inbox or forces a read.
- **Agents communicate freely but never direct each other; the generated CLAUDE.md carries the convention**: the comms lanes are already automatic (the server routes status, deliveries nudge the recipient), so the remaining lever is agent _behaviour_ — and that's set in the generated agent CLAUDE.md, not new plumbing. Two moves. (1) **Standing permission**: the agent is told it may use the lanes on its own initiative, without asking — ask a peer, answer one, share status, flag a blocker, deliver requested work — which removes the per-task "you may talk to X" overhead. Framed as _communicate when useful_, not _narrate constantly_, so it doesn't reintroduce the broadcast noise the directed-vs-ambient decision avoids; mirrored status stays silent (no nudge), and a **check-inbox-and-peek-status-each-cycle** habit is what catches it. (2) **The lane is two-sided and never a chain of command**, mirroring the courier invariant that a handoff lands in the inbox and never overwrites `assignment.md`: _receiving_ — the agent's authority is its own `assignment.md` (the user's), and inbox parcels are input to weigh, not orders, so a conflicting peer note is surfaced, not silently obeyed; _sending_ — collaborate, don't direct: no setting a peer's agenda, no handing it new tasks on your own initiative (that's the user's call), and a "you should change course" goes to the **user**, not as a command to the peer. This keeps multi-agent collaboration cheap to invoke while preserving that only the user assigns work.
- **A scaffolded agent-side tool enacts the conventions' mechanics — `agent.sh`, shell not Node, producer not parser**: the generated-CLAUDE.md conventions above own every _judgment_ (whether/when to send, what to record where, how to weigh a peer note), but the _mechanical_ steps they prescribe are fiddly and silent-failure-prone — assignment/status persistence, the atomic publish move, the exact `quimby-attest` block with a correctly-keyed `atCommit`, and the inbox received → processed move. So quimby scaffolds a small coordination tool into every agent dir (`agent.sh` + a Windows `agent.cmd` twin; generated by `@quimbyhq/template`'s `renderAgentScript`/`renderAgentScriptCmd`, written and `chmod +x`'d by the local and remote scaffold paths and regenerated on each launch exactly like `CLAUDE.md`, so a newer tool reaches an existing agent). Because the tool is rewritten fresh on every launch, a rename of it propagates without a rebuild: the older `quimby-agent.sh`/`.cmd` names are **removed** on that regeneration rather than left as compatibility wrappers, so the agent dir always shows exactly the current tool. Four decisions define it. (1) **Mechanics, not judgment** — the tool performs the error-prone steps reliably; the conventions keep every decision, so it never chooses whether to send, only how. (2) **It never crosses the boundary** — `agent.sh` only ever writes the agent's _own_ workspace; carrying a parcel to a peer or out to the user's repo stays the host's job (`quimby dispatch`/`merge`), preserving the courier's boundary. (3) **A POSIX shell script, so it adds zero remote dependency** — it runs where the agent runs, and quimby's guaranteed remote floor is `sh` + `git` + coreutils (git is already a hard remote dep) while Node is **not**: a cheap Ollama/qwen runtime may have no Node, and cheap agents are the point, so a Node tool would either exclude them or force a new remote dep. The `.cmd` twin covers the lone non-POSIX case (a Windows-hosted local agent); it is best-effort and unverified, since every remote sandbox is POSIX and the `.sh` is canonical. (4) **An optimization, never a replacement, and DRY where duplication is fragile** — the host is always the canonical _parser_ of these formats (TypeScript), the agent only ever the _producer_, so there is no second parser to drift, just a producer/consumer contract pinned by a round-trip test; and an agent whose runtime lacks `sh`, or that never got the script, still works by following CLAUDE.md. A Node CLI shipped into the sandbox, or teaching the agent to shell out to the host `quimby`, were both rejected: the first breaks the no-new-remote-dep floor, the second breaches the boundary the courier model rests on.
- **The diff is the wire format across the boundary**: A sandboxed/SSH agent is not a reachable git remote, so the host cannot `git fetch` it. Carrying the diff is what makes cross-agent and agent→host movement possible at all.
- **Squashed merge by default**: Agent commit history is useful context but shouldn't leak into the real repo. The boundary ensures the user curates what enters.
- **Merge is a merge, not a patch**: The agent's diff is reconstructed on a temp branch from the seed, then merged into the target — not patched directly onto the working tree. A patch fails when the target has moved past the seed (the common case with multiple agents); a merge handles it with git's 3-way machinery, producing standard merge conflicts the user can resolve with tools they already know. The agent is never touched — no non-working state inside a sandbox. Conflicts live on the host where the user has full tooling. The merge is a **plain, fast-forward-when-possible** merge, deliberately **not** `--no-ff`: when the target hasn't moved past the seed the agent's commit fast-forwards into clean linear history; a merge commit appears only when the target genuinely diverged, and then it is a standard `Merge branch …` node the user recognizes and can rebase away. Forcing `--no-ff` (with a parcel-derived merge message) was rejected because it stamped a same-message work-commit + merge-commit pair onto history for every merge — visual duplication with no upside for a user who prefers the landed work to read as the newest commit.
- **Merge advances the seed by default, but only when it's lossless**: an agent's diff is cumulative (working tree vs `quimby/seed`), so the iterate-one-agent loop — merge, then ask for a revision — otherwise re-carries already-shipped work every round and conflicts the moment the regenerated diff isn't byte-identical to what landed. So a clean, committed `merge` advances the agent's seed onto the merge (`sync -f`), making the next diff incremental; `--no-sync` opts out (and `--sync <ref>` advances _and_ retargets the agent's sync ref to `<ref>` — see the `--sync` decision below). It is deliberately **conditional**, not a blind default, because the advance is a hard reset: it runs only when the merge settled onto the branch the agent tracks (its `syncRef` — not a `-b` landing branch or foreign `-t`, which are deliberate deferrals) **and** the agent hasn't moved since the snapshot the merge captured (equal content hashes ⇒ identical tree ⇒ nothing to lose). Either condition failing leaves the seed untouched with a pointer to the manual `sync --current -f`. This mirrors `assign`'s sync-by-default naming, but the two aren't symmetric: `assign`'s pre-sync is the non-destructive rebase (keeps work), so it can default on freely; `merge`'s post-advance is the destructive reset, so it earns default-on only through those conditions. Only a **fully-committed landing** advances; any uncommitted landing (`--patch`, the no-TTY degrade, a `--commits` remainder) settles nothing, so the seed stays put. Whenever the seed is _not_ advanced but a standard merge would have, `merge` prints the catch-up sync so the agent never silently goes stale — surfacing the staleness at the one moment it's created rather than at the next conflicting merge.
- **Merge conflict recovery leans on git, with silent staging auto-heal — no `merge --continue/--abort`**: a conflicted `merge` is left in progress for the user to finish with native `git merge --continue`; quimby adds no wrapper command for the in-progress case. For the **abandoned** case (`git merge --abort`, which leaves the staged parcel orphaned), quimby doesn't ask the user to remember a cleanup verb either — the next `merge`, when it stages fresh work, **silently clears** a leftover staging area, gated on there being **no git merge in progress** in the target (an in-progress merge is the live retry path, so its parcel is preserved). Merging an already-staged parcel _by name_ skips the heal, since that path points at staging deliberately. A dedicated `quimby merge --continue/--abort` or a `gc` verb was rejected as ceremony over an auto-heal that costs the user nothing to remember.
- **The boundary never fabricates a commit message**: crossing into the real repo is an explicit, curated act, so the one commit a squashed `merge` creates is authored by the user — git's editor opens prefilled with the agent's own subject (`-m` skips it). `--commits` carries the agent's real commit messages verbatim and leaves any uncommitted remainder loose in the working tree (the agent didn't commit it, so quimby doesn't invent a commit boundary it never drew); `--commits -m` opts that remainder into one named commit. The rule holds even non-interactively: with no `-m` and no TTY there's no one to author the message, so rather than stamp a generic `Merge work from <agent>`, squashed **degrades to `--patch`** — the work still lands, uncommitted, message printed for the user to commit with. This collapses the three modes onto one rule (a landed commit is either the agent's or yours, never quimby's) and makes `--patch` legible as "what squashed becomes when the message can't be authored." The celebratory quip fires only on a fully-committed landing, so success reads as success and an uncommitted landing reads as "more to do."
- **Advisory checks are cooperative — ask, attest, relay; never execute, never gate**: quimby cannot honestly run an agent's build/test check, because it is a host-side courier running _outside_ the sandbox where the deps live. So quimby _asks_ the agent to self-check (`nudge --verify`, `assign --verify`, an agent `verifyByDefault`, or a CLAUDE.md convention), the agent runs its own `check` command (`set <agent> --check`) inside its sandbox and records an **attestation** — a `quimby-attest` block in `status.md` (`command`/`result`/`summary`/`atCommit`) — and quimby **relays** that self-report at the boundary: shown in `status`, printed before `merge`/`handoff`, later embedded in the parcel `meta.yaml`. Quimby never re-runs the check and **never gates** — a failed/absent/stale attestation warns, the human still decides. Because it is a self-report relayed, not a quimby guarantee, the display headline speaks the CLI noun **check** with a **(self-reported)** qualifier carrying the relay honesty — never "verified"/"validated", which would imply quimby ran the check itself. This is the only honest shape: the check runs where the deps are, and the boundary stays the human's call.
- **Assign syncs by default, but never loses the message**: Assigning to a stale agent wastes tokens — the agent works off wrong files, discovers issues that are already fixed, or produces diffs that conflict on merge. `assign` writes `assignment.md` first (the user's intent is durable), then syncs the agent to its base. If sync fails (rebase conflict), the assignment is still on disk — the user's message is never lost — but the nudge is suppressed (don't wake an agent on a stale baseline). The user resolves the conflict and runs `quimby sync` manually. `--no-sync` opts out entirely. `--sync` is a **ref string** here too: `--sync <ref>` retargets the agent's sync ref to `<ref>` and syncs onto it before the assign (citty yields `false` for `--no-sync`, a string for `--sync <ref>`, so one flag carries both the skip and the retarget).
- **One `--sync`, optionally-valued and uniform — not a `--sync`/`--advance` split**: `--sync` used to mean two things — a ref-string on `add`/`set` (the sync root) but a plain boolean on `assign` and `merge`. Rather than split it (an earlier take renamed merge's boolean to `--advance`, which added a second verb to learn), it is kept as **one flag** made **optionally-valued** and given **one meaning everywhere**: bare `--sync`/default is the natural sync (`assign` syncs to the agent's `syncRef`; `merge` advances the seed onto the landed branch), `--sync <ref>` names/retargets the sync ref (`add`/`set` set it; `assign` retargets-then-syncs; `merge` advances **and** retargets `syncRef` to `<ref>` — the post-advance already _is_ a sync to the landed branch, so a ref just points it elsewhere), and `--no-sync` skips. This leans on citty distinguishing "flag present, no value" (bare, `''`) from "present, with value" (the ref) from "negated" (`--no-sync` → `false`); `-s` stays the short form on `add`/`set`. One flag, one mental model, optionally carrying a ref — no new `--advance` verb.
- **Server is infrastructure, not convenience**: Agents in sandboxes can't see each other. The server is the only entity with cross-agent visibility. It's architecturally necessary, not a nice-to-have.
- **Status mirrors to every agent; discernment moves to the reading side (pull-on-demand), replacing subscribe**: the server used to route each agent's status only to _subscribers_ (`subscribe`/`unsubscribe`, a `state.subscriptions` map). That was dropped for **always-mirror**: the server writes each agent's `status.md` into _every_ other agent's `status/<name>.md`. The key realization is that the token cost was never the _mirroring_ (status files are tiny — disk, not tokens) but the _reading_. Subscribe bounded reading indirectly, by bounding what got mirrored; always-mirror + **pull-on-demand** bounds it directly — agents don't slurp the whole roster each cycle, they use `./agent.sh peers` and read a peer only when they have a reason (the generated context says so). So mirroring can be universal at near-zero cost, the "forgot to subscribe" silent miss disappears, and the `subscribe`/`unsubscribe` verbs, the state field, and the server's POST/DELETE endpoints all go away. The asymmetry that makes this right: the **human** has unbounded context and sees every agent on demand via `quimby status`; the **agent** is token-constrained, so it gets availability (everything mirrored) with bounded attention (read on demand). The courier split stays sharp: **directed** work is pushed (`handoff` + nudge); **ambient** status is pulled (a peer-status peek) — folding status into re-sent parcels would be chattier and lose the publish-once/pull property.
- **Auto-dispatch enacts the outbox**: while running, the server carries settled outbox drafts to their recipients on the poll cycle — the final removal of the human-as-messenger from the directed channel, since `dispatch` was already agent-authored routing the human merely triggered. It is orthogonal to status mirroring: `dispatch` carries **directed, discrete parcels** (a diff and/or note addressed to one recipient), status mirroring reflects **ambient status**. The safety comes from the same courier invariants automation needs: content-hashed names dedupe, bounces stay queued, `out/sent/` is the ledger — plus the **atomic author-then-publish** (draft → queued via one `mv`) that keeps a half-written parcel out of the scanned queue by construction, a **settle debounce** as the fallback (a queued parcel carries only after its files are unchanged for a full cycle), and an **attempt-once** rule so a bad address never nudges in a loop. Because human-gated dispatch was also the accidental rate limiter against the #1 pain point (token exhaustion), auto-dispatch is opt-out per server run (`--no-dispatch`), never per-agent config.
- **`serve -it` binds the server's life to a terminal, like `docker run -it`**: the server is a background daemon, but running it in one terminal and typing commands in another is friction. `-it` starts the server and stacks a shell on top in the same terminal, so commands run live against the server underneath and the server dies when you leave the shell. Ctrl+C is deliberately **not** a single-press kill — the shell owns it for interrupting commands, and only `exit`/Ctrl+D or a quick **double** Ctrl+C tears the server down, so interrupting a long command never accidentally takes the server with it. The server library returns a `stop()` handle rather than installing its own `process.exit`, so the command layer owns lifecycle in both the plain and interactive modes.
- **`serve --stop` is a flag, not a `stop` subcommand**: a background `quimby serve` (plain, no `-it`) needs a way to be stopped from another terminal without hunting the pid. That lives as `serve --stop` rather than a top-level `quimby stop`, because **`quimby stop` is already the agent verb** (kill an agent's tmux session) — a second meaning would collide. `--stop` reads `server.json`, signals the pid (the server's own SIGTERM handler runs the graceful `stop()`), and removes the pidfile so a stale entry never lingers; a missing/dead server reports cleanly and still sweeps any stale pidfile. The `stopServer` primitive lives in the server package beside `getServerInfo`, so the CLI stays a thin caller.
- **Three interaction modes coexist**: Interactive (run), headless (start), and server (serve) are separate concerns. run/start manage individual agents; serve manages the connections between them.
- **Stable IDs, not names**: `QuimbyState.id` and `AgentState.id` are UUIDs generated at creation and never change. tmux session names are derived from IDs, so renaming an agent doesn't orphan a running session.
- **One UUID is an agent's identity across every surface; the name is only a display label**: an agent's `id` keys its on-disk directory (`.quimby/agents/<id>/`, local and remote), its tmux session (`qb-<projectId[:8]>-<agentId[:8]>`), and its sandbox name. The friendly name is a pure display attribute — stored in `state.yaml`, shown in `quimby list`, and set as the tmux **window** title (`rename-window`) on each `run` so the on-screen label tracks renames. `quimby rename` also pushes the new label onto the agent's **live** window immediately (`renameAgentWindow`), so a running or attached session — and any dashboard tab sharing that linked window — updates in place instead of waiting for the next `run`; a stopped agent simply no-ops. Because nothing on disk or in a live session is keyed by the name, **rename is a pure relabel**: the directory never moves (no `fs.rename`/`mv`), so the sandbox and tmux session bound to that path survive, and no running work is orphaned. Existing name-keyed workspaces migrate on state load (local: `fs.rename` to the id-dir; remote: a guarded `mv` on next `run`).
- **The sandbox name is a path-and-command hash over the UUID-keyed directory**: `sbx` is path-sensitive — it persists the working directory's absolute path behind its `--name` — so the name must track the path, not pin it. It also refuses to reuse a named sandbox for a different base command (for example a sandbox first created for `claude` cannot later run `codex`), so the runtime passes `--name qb-<agentId[:8]>-<hash>`, where the hash covers the project id, agent id, agent directory, and base entrypoint command. Since the directory is UUID-keyed, the path portion is **stable across a rename**, so a same-command sandbox is reused; a genuine **relocation** (the `.quimby` tree moving, or a clone mounted at a different path) changes the absolute path, flips the hash, and lets a fresh sandbox fall out instead of a stale name pointing at a directory that is gone. A real runtime-profile command change (`claude` → `codex`) also flips the hash, avoiding `sbx`'s "already exists and runs ..." collision. This is why a path-and-command hash beats keying the sandbox on the bare UUID: a UUID name would survive rename but go stale on relocation and collide across command changes. The friendly name is deliberately **not** in the sandbox name — it changes on rename and would needlessly break reuse; the `agentId[:8]` prefix is a stable, greppable handle for `sbx ls`. `run` and `exec` still share a sandbox when they use the same base command.
- **Runtime profiles are named launch/provisioning intent, not secrets**: roles answer "what kind of agent is this?"; runtime profiles answer "how should that runtime be launched and checked?". An agent stores only a profile name (`defaults.runtimeProfile`), while layered config resolves the profile at launch/doctor time. When a repository tracks `quimby.yaml`, that file is authoritative for the shared profile/role fields it defines (notably `runtime` and `entrypoint`) because it is visible and reviewable; ignored `.quimby/local.yaml` and user config add local-only names and fill private machine details such as provider hosts, host aliases, env, and credentials-adjacent settings. This keeps the shared launch graph auditable without forcing worker names, IPs, or auth-sensitive values into git. The resolver is its own package: config loading stays dumb, adapters stay focused on command shapes, and both launch and doctor consume the same small selection object (`runtime`, `entrypoint`, `env`, required tools). For SSH agents, those profile requirements are checked on the destination host, because that is where `openshell`, `ollama`, `sbx`, and their auth state actually live.
- **An agent stores its role and resolves launch config fresh; drift is detected on the resolved command, and `restart` applies it**: an agent used to bake its role's runtime profile/entrypoint into a flattened `defaults` at creation, so editing the preset/role/profile later — or _renaming_ a profile — left existing agents pointing at a now-stale (or dangling) name, and a running tmux session kept whatever command it was born with. Two coupled fixes, mirroring the host-alias pattern. (1) **Store the role reference, resolve at launch**: `AgentState.role` records the config role, and `resolveAgentLaunchDefaults` resolves runtime profile/entrypoint from _current_ config through it at every launch (an explicit `--runtime`/`--cmd`/`--runtime-profile` still wins; a role that no longer resolves degrades to the stored `defaults`, never a hard failure). So a profile edit or rename propagates to existing agents with no rebuild — `quimby set <agent> --role <role>` attaches a role to a legacy agent to opt it in. (2) **Drift is the _resolved command_, not the name**: on session create, `launchFingerprint` (a hash of the resolved `runtime`+`entrypoint`) is stored in `AgentState.launchedWith`; `run`/`start` compare it against the freshly-resolved command and _warn_ only when they differ — so a role rename that resolves to the same command leaves the session untouched (no false drift), and a real change surfaces as an advisory pointer to `quimby restart`. Restart is a **dedicated, explicit verb** (`quimby restart <agent…> [--all]`), never automatic and never `run --restart`/`start --restart` (a preset user drives the whole fleet, and a magic auto-restart would silently discard the live session's chat context): it kills the session and relaunches through `start`, so the agent comes back on current config and resumes from `status.md`. The invariant: you only ever lose live context when the command genuinely changed _and_ you chose to restart — the retained-session model is preserved, drift is surfaced, and applying it is one deliberate command.
- **A host alias is a first-class indirection bound per-machine, never a flattened address**: a tracked `quimby.yaml` can _declare_ a host alias (`hosts: { remote: { host: remote } }` — or simply omit `host`) without committing a private address; the concrete `user@host` is bound per-machine into ignored `.quimby/local.yaml` (default) or user config (`--global`), so it never touches git. The distinction is explicit: an alias whose `host` is absent, empty, or **equal to its own name** (the self-referential placeholder) is **unbound**. Crucially the agent stores the alias _reference_ (`location: { type: ssh, alias: remote }`), not a flattened address — the address is resolved from layered config **at launch**, so binding it once (or rebinding it) propagates to every agent that shares the alias, and a `restore` on a fresh machine reconnects by re-resolving rather than carrying a stale IP. An earlier design flattened the alias to a concrete `location.host` at `add`/`restore` time, which killed the indirection: the stored string was handed straight to `ssh`, `run` never re-consulted config, and a nuked-then-restored workspace baked in a bogus `host: remote` that failed DNS with no way to bind it. Resolution is a pure function (`resolveSSHConnection`), so it also recognizes a **legacy** flattened `host` that happens to name a declared alias and treats it as that alias — the migration is behavioral, not a state rewrite. When an unbound alias is actually needed (a launch, or `restore`'s own scan connection), quimby **prompts once** and persists the answer (interactive only — a non-TTY caller gets a `quimby host <alias> --set …` hint instead of a mystery DNS error), then continues; sibling agents on the same alias resolve without re-asking. Prompting lives in the CLI (`@clack/prompts`) while the resolver/persistence stay pure in `@quimbyhq/workspace`, keeping packages clack-free. `quimby host` is the explicit, scriptable counterpart — list aliases with bound/unbound status, or `--set`/`--global` to bind without launching. The rule mirrors the boundary invariant one level down: the tracked repo declares _intent_ (an alias exists), the private layer supplies the _secret_ (its address), and quimby fills the gap the moment it bites rather than failing opaquely.
- **"Presets" name a saved arrangement; a "default" preset makes bare `quimby up` and `quimby run` do the obvious thing**: the config concept that names a set of agents + a layout was renamed `recipes` → **`presets`** to avoid the cooking metaphor and, more importantly, to dodge the two collisions the obvious alternatives carried — "workspace" already means the `.quimby` project/state/durable-storage (a whole package, `resolveWorkspace`, `~/.quimby/workspaces/…`), "profile" is taken by `runtimeProfiles`/the `runtime-profile` package, and "template" by `@quimbyhq/template`. A legacy `recipes:` key still loads (folded into `presets`, presets winning a clash), so existing configs keep working. Separately, a top-level `default:` names the preset a bare `quimby up` (no preset) creates and a bare `quimby run` (no agent, no `--layout`) opens — and `quimby run --layout <name> --default` records it. The run-target model that shapes this: a preset is **not** parallel to an agent, it's parallel to a **layout** (agents/groups compose inline into a layout; a preset contributes its saved layout), so `run` opens either agents (composable positionals) or one named/inline layout — the positional stays unambiguously agents/groups, and a saved layout/preset is reached explicitly via `--layout`, with `default:` the only thing that makes the empty invocation meaningful. The default is persisted under the same rule as host bindings — auto-saved to ignored `.quimby/local.yaml` (or user config with `--global`), hand-settable in a tracked `quimby.yaml` for a team-shared default but never auto-written there.
- **A service is a dashboard-local command pane, not a retained session — so it stops on exit by construction**: a layout can place a named host command with a `$name` token (`services: { server: "quimby serve" }`, then `(host $server)`), giving a "cockpit" dashboard where one pane runs the server (or a log tail, `watch`, etc.) beside plain shells. The token grammar extends the existing host slot: bare `$`/`host` is an interactive shell, `$name` runs `services.name`. The lifecycle question — should the command survive a dashboard rebuild? — is answered by _where_ the pane lives: a service is a window in the **ephemeral view session** (exactly like a `host` shell), not a retained agent session, so when the dashboard is torn down (any exit, or the rebuild on the next `quimby run`) the pane dies and the command with it. This is deliberately the opposite of an agent (retained across dashboard rebuilds): an agent is durable work you reconnect to, a service is scaffolding around this viewing of it. It means a service like `quimby serve` **starts with the dashboard and stops when you leave** rather than leaking a background process — at the cost of restarting on each `run`, which is why `serve` should additionally be made idempotent (detect a live server and no-op) so an overlapping run never double-starts it. A retained-service model (a service that outlives the dashboard, link-windowed like an agent) was considered and rejected as the default: it reintroduces exactly the "who stops the server" bookkeeping the ephemeral pane avoids, and the stop-on-exit behavior is what the cockpit use-case actually wants.
- **Runtimes validate before launch and tear down their sandbox on removal**: a `RuntimeAdapter`'s `setup`/`teardown` are no longer stubs. `setup` **validates the runtime CLI is installed** — `prepareLocalTmuxLaunch` calls it before any tmux work, so launching an `sbx`/`openshell` agent without the tool fails with a clear "not on your PATH — install it, or use `-r local`" instead of the agent's pane dying the instant it execs a missing binary. Only a missing binary (`ENOENT`) counts as absent; a quirky `--version` is assumed present rather than wrongly rejected. Validation is **local-only**: an SSH agent's runtime CLI lives on the remote host, so `prepareSshLaunch` skips the local probe (checking the wrong machine would be worse than not checking). `teardown` removes the agent's per-agent sandbox (named by the same UUID-keyed path hash as `run`) when `quimby remove` deletes a local agent — best-effort, swallowing failures so cleanup never blocks removal, and skipped for SSH (that sandbox is remote). The exact `sbx`/`openshell` removal verb is best-effort and may need adjusting as those CLIs settle.
- **Provisioning adopts-or-creates keyed on git origin; `restore` is the escape hatch, not the happy path**: the project UUID that namespaces the remote workspace (`~/.quimby/workspaces/<projectId>`) lives only in the ignored `.quimby/`, so losing that directory (fresh clone, `git clean -x`, new machine) used to force the user through `quimby restore` — a manual, alias-and-`--id` archaeology step — before they could run their configured workspace again. Worse, if they _didn't_ restore, the next `add`/`run`/`up` minted a **fresh** UUID and silently orphaned the old remote agents as unreachable garbage. Both problems have one root fix: make workspace resolution (`resolveWorkspace`/`ensureWorkspace`) **adopt-or-create** instead of create-blindly. Before minting a fresh id it (1) relinks local durable storage on a registry match, then (2) scans every **bound** host alias in config for a remote workspace whose git origin matches this repo — exactly one match is **adopted silently** (reusing its id + roster + remote clone), zero creates fresh, more than one throws for the user to pick (`restore --host <alias> --id <id>`). This makes remote provisioning **idempotent on git origin**, which is the anti-garbage guarantee: a duplicate workspace can no longer appear by accident. Three deliberate constraints keep the magic safe. **Bound aliases only** — silent adopt never prompts for an address (prompting mid-`run` is exactly the friction being removed); an unbound alias falls through to the existing explicit paths. **Unreachable is skipped, never fatal** — a dead VPN or sleeping host (or a probe past `QUIMBY_REMOTE_PROBE_TIMEOUT_MS`) degrades to "no match," so an offline command doesn't hang or crash; a purely-local project with no bound alias never touches the network at all. **`restore` stays** as the explicit, scriptable escape hatch for what auto-adopt won't guess (a different alias, cross-machine recovery, `--id` disambiguation) — it and the silent path now share **one** reconstruction core (`reconstructRemoteWorkspace` in `@quimbyhq/workspace`), so both produce byte-identical recovered state. This required `@quimbyhq/workspace` to gain a `@quimbyhq/transport` dependency (allowed by the DAG: workspace sits above transport), with the clack prompt kept in the CLI so the package stays prompt-free. The GC half is deliberately **adopt-only, never auto-delete**: since duplicates can't accrue going forward, cleaning up ones the old behavior already made is an explicit `quimby storage prune-remote --host <alias>` — it previews (or with `--force` removes) only the remote workspaces for _this_ repo that are _not_ the active id, never touching another repo's lanes or the workspace you are on, and refusing outright when there is no local workspace to identify the one to keep. A "prune on reconnect" default was rejected: deleting remote state during a routine `run` is exactly the surprise the boundary philosophy avoids.
- **SSH lazy init**: SSH agents are not set up remotely at `quimby add` time. The remote clone, tagging, and scaffolding happen on first `quimby run`. This allows adding SSH agents without an active SSH connection.
- **rsync as transport**: SSH agents sync the project source via rsync before each run. The remote clone is a local clone of the rsynced source tree — no direct git remote needed on the agent side.
- **User config/data roots follow the host platform, project state does not**: project-local control state stays at `.quimby/` because it is intentionally visible at the repo boundary. User-level config and durable workspace storage are different: they survive `git clean` and should feel native on the host. Linux defaults to XDG (`~/.config/quimby`, `~/.local/share/quimby`), macOS defaults to `~/Library/Application Support/quimby`, and Windows defaults to AppData (`Roaming` for config, `Local` for data). `XDG_CONFIG_HOME`, `XDG_DATA_HOME`, and `QUIMBY_DATA_HOME` still override those defaults so power users can pin storage explicitly.
- **tmux is the universal substrate; the dashboard is a viewport**: every agent — local or SSH — always runs in its own persistent tmux session (no foreground path). Disconnecting doesn't kill the agent. `quimby run <agent>` does `tmux new-session -A` against the stable-ID session name, so it **grabs the one canonical session wherever it is launched** (started here, by `start`, or shown in a dashboard) and creates it only if absent; the window label is refreshed to the display name on every (re)attach so it tracks renames. Running several agents (`quimby run <a> <b>`) opens a **dashboard** — a stateless viewport session that `link-window`s each local agent's own window in as a tab (SSH tabs ssh-attach the remote session; `host` is a plain shell). link-window (not nested `attach`) is deliberate: the agent's window is _shared_, so the dashboard tab and a direct `quimby run <agent>` are two live views of the same window, yet only the dashboard's single status bar shows — no nested inner status bars, and nothing set at the dashboard's session scope alters the agent's own session. Because the dashboard only borrows a view, the per-agent sessions outlive it, so it is torn down and rebuilt on each run (dodging tmux's "duplicate session"), tabs are numbered from 0 to match `Ctrl-b 0/1`, and per-tab activity/silence highlighting is set as window options — shared with the linked agent session but invisible there, since that session's plain `window-status-format` doesn't react and visual-* stay off.
- **No raw-SSH fallback for retained agents**: SSH agents still require remote tmux even when their repo already exists. A raw `ssh <host> <entrypoint>` fallback would appear to work but would lose the properties users rely on: reconnectable sessions, `start`/`stop`, `nudge`, dashboards, and stable session names. So `prepareSshLaunch` checks remote `tmux` on every launch and fails before constructing the session if it is missing; first-run provisioning still checks `git`, `rsync`, and `tmux` together.
- **Quimby owns its tmux, on a dedicated server**: rather than depend on whether the user has a good `~/.tmux.conf`, Quimby runs all its tmux on an **isolated server socket** (`tmux -L quimby`) started from a **bundled config** (`-f .quimby/tmux.conf`, generated by `renderTmuxConfig`; the remote twin is written over transport). The isolation is the point: Quimby's settings never leak into the user's default server, and its sessions never clutter the user's `tmux ls`. The config is **layered** — calm aesthetic defaults (a muted status bar showing the agent/window name, instead of tmux's default bright-green bar) → `source-file -q ~/.tmux.conf` so the user's own keybindings/theme still apply → Quimby's functional must-haves set _last_ so they win: true-color passthrough (`terminal-overrides ",*:Tc"`), `mouse on` (scroll wheel reaches scrollback — default tmux needs copy-mode), a generous `history-limit`, and stable window names. This fixes what session-scoped options couldn't: true-color (needs `default-terminal`/overrides at server start) and scrollback depth (needs `history-limit` before the pane is created). Every quimby tmux call — `new-session`, `has-session`, `send-keys` (nudge) — targets `-L quimby`, so they all see the same sessions. Trade-offs: `-f` is read only at server start, so config changes apply after the quimby server is next empty and restarts; and with mouse mode on, click-drag selects into tmux's copy buffer (hold Shift for the terminal's native selection). To reach a session by hand, use `tmux -L quimby attach` (but `quimby run` handles attach for you).
- **Nudge: wake a live agent in place, policy per movement**: writing `assignment.md` or dropping a parcel in an inbox is silent — a running interactive agent won't notice until its next prompt, leaving the user to switch terminals and type by hand. So quimby can inject `<text>` + Return into the agent's tmux session (`tmux send-keys`, keyed by the agent's stable UUID, so a rename never loses it). The default policy follows how directed the movement is: **assign** always nudges (you named one agent and gave it a task), **dispatch** nudges by default (the boss agent authored an addressed note with intent), **handoff** nudges only when the parcel carries a **note** — the instruction half — since a handoff is often pure data (a diff with no note); `--nudge`/`--no-nudge` force either way. The standalone **`quimby nudge <agent> [-m]`** is the explicit, ad-hoc live channel: no `-m` types `"continue"` (the lightest kick), and `-m` types arbitrary text verbatim — including CLI control commands like `/clear` or `/model …`. The text is the wake-up; the durable work still lives in `assignment.md`/the inbox, so a missed or skipped nudge only delays, never loses. A local non-tmux agent runs in the foreground (the user is already attached), so there is nothing to wake. There is deliberately **no separate `send` verb**: a first-class "send the agent instructions" command would compete with `assign` for the task-giving slot and tempt users onto an ephemeral channel when they need a durable task; folding arbitrary/control sends into `nudge -m` keeps the capability without advertising a verb that cannibalizes `assign`. Durable task-of-record is `assign`'s job; `nudge` is the live, ephemeral poke. **`nudge --all`** broadcasts (e.g. `--all -m "/clear"`) but first **probes tmux for live sessions** (`has-session`) and targets only those — so it never spams dead agents, and the logged target list shows exactly which live sessions received it.
- **Headless = detached tmux + nudge; `start`/`stop` bracket it**: the headless mode fell out of two pieces already built. A tmux session can be created **detached** (`new-session -d`) — running but attached to no terminal — and `nudge` can already type into a session by its stable UUID. Detached session + nudge _is_ headless execution: the entrypoint runs unattended and picks up work when woken, no terminal held. So `quimby start` is `run`'s launch with `-A -d` instead of an attach (the `-A` keeps it **idempotent** — starting an already-running agent is a no-op, never a second session), and `quimby stop` is `tmux kill-session`. `start` shares `run`'s whole launch path (SSH sync + lazy init, runtime spec, bundled tmux config) via a common `launch` helper, differing only in attach-vs-detached. Because a detached session is inherently a tmux session, `start` **enrolls a local agent into tmux** (persists the `tmux` flag) if it wasn't — otherwise `run`/`nudge`/`list` wouldn't recognize the session it just made. `stop` warns before killing an **attached** session (a client is in `run`), and both commands are no-op-safe (start-when-running, stop-when-stopped just report). The durable work is on disk (`assignment.md`, the repo), so `stop` only ends the process — never the work.
- **The recovery loop: a fresh launch resumes from its predecessor's `status.md`**: `status.md` is framed as an agent's handoff to its own successor, so a crashed or context-reset agent should pick up from it rather than starting cold. Two layers make that automatic. Passive (both `run` and `start`): the generated CLAUDE.md's first work step tells the agent, "if `status.md` is non-empty, a prior instance left it — read it first." Active (`start`, the headless case where no human is watching): because `start` only ever creates a _fresh_ session (it returns early when the agent is already up), a non-empty `status.md` means there's something to resume, so after a short settle (the new pane needs a beat to bring up its prompt) quimby types a one-line resume request pointing at `@status.md`. `run` leans on the passive layer alone — its human is attached and can see the status — rather than restructuring its blocking attach to inject a nudge. An empty/absent `status.md` resumes nothing, so no nudge fires.
- **The agent keeps `assignment.md` current from direct user input — closing the recovery loop's authority gap**: the recovery loop makes a successor resume the right _place_ (`status.md`), but the successor also treats `assignment.md` as its authoritative _task_ (so a peer can't hijack it). Those two facts combine into a trap: if the user retasks the agent by **typing into its live session** (not via `quimby assign`), the new task lives only in volatile context — `assignment.md` still holds the old one — so a crash/`/clear`/reset resumes the successor confidently on the **wrong** task. Only the agent can see a directive typed into its own session, so only the agent can durably record it; the fix is a **generated-CLAUDE.md convention** (behaviour, not plumbing), mirroring how the comms conventions live there. The rule discriminates by whether input redefines the task — the test is _"would a fresh instance with only `assignment.md`+`status.md` pursue the wrong goal without this?"_: a changed goal/deliverable/scope/hard-constraint is rewritten into `assignment.md` (a clean snapshot, not a changelog), an approach/context change goes to `status.md`, and transient/local input ("check line 40") is just acted on. Bias is **toward recording** when unsure (a successor doing the wrong task is worse than a slightly over-specified assignment), and it is the **user's channel only** — a peer inbox note is never an assignment. These writes are **quiet** (no "I updated status.md" narration): `status.md` is a continuous journal and the user reads state on demand via `quimby status`, so the writes are observable-on-pull, not push-narrated. A host-side `nudge --assign` (write-then-send, for retasking from the host) was considered and deferred — the convention covers the in-session case, which is where the gap actually bit.
- **`list` shows live session state, probed from tmux**: the roster reports each agent as **running** (a detached headless session), **attached** (a client is in `quimby run`), or **stopped** (no session) — read straight from tmux's `#{session_attached}` count, the source of truth, not from any stored flag. Crucially the probe **gates on `has-session` first**: `display-message` against an unknown target silently falls back (empty output, exit 0) whenever the quimby tmux server is up for _other_ agents, so without the gate a stopped agent would misreport as attached. A local non-tmux agent has no session to probe and reads as stopped even while a foreground `run` is live — an accepted limitation, since a foreground agent is one the user is already sitting at and there is no detached session to wake or kill.
- **Aggregate remote status is bounded, not exact-at-any-cost**: `list` and the status overview are scanning surfaces. They should not hang behind a dead VPN, sleeping SSH host, or wedged remote tmux server. Remote probes for session state, pending work, and sync status are therefore bounded by `QUIMBY_REMOTE_PROBE_TIMEOUT_MS` (default 5 seconds; `QUIMBY_REMOTE_STATUS_TIMEOUT_MS` is accepted as an alias). A timeout uses the same conservative fallback as an ordinary probe failure (`stopped`, no work summary, no behind count) and marks the row with `remote timeout` so the user knows the absence of detail is a probe failure, not a clean state.
- **The session transcript is written by the pane itself, via `pipe-pane`**: tmux scrollback is bounded (`history-limit`) and dies with the session, so a durable record needs a file. Rather than bolt a separate logging step onto `run`/`start` (whose attach is a blocking call, awkward to sequence a post-create `pipe-pane` around), the launch **shell command the pane already runs** — the one that does `rename-window` — also runs `tmux pipe-pane -t "$TMUX_PANE" 'cat >> session.log'`. Targeting `$TMUX_PANE` needs no session lookup, and because the command runs once per pane lifetime (fresh session or respawn) it opens the pipe exactly once for both the attached (`run`) and detached (`start`) paths, with no extra tmux round-trips. `quimby log --follow` then `tail -f`s the file. Local agents only for now — an SSH agent's transcript lands on the remote host, which the follow path doesn't yet reach.
- **Transport abstraction**: All agent I/O goes through `LocalTransport` or `SSHTransport`. Commands don't need to know where an agent lives — they call `transport.exec`, `transport.writeFile`, `transport.rsyncTo`, etc.
- **Transport never commits**: Work is captured from the agent's working tree (committed + uncommitted + untracked) via a throwaway index, so `handoff`/`dispatch`/`sync` never make a commit in an agent — frequent use never litters its history, and agents needn't use git at all. The only commit is the optional one at `merge`, the boundary.
- **Three levels of "catch up"**: `sync` keeps the agent's work (rebases it); `sync -f` drops the work but keeps the agent (mailbox intact); `rebuild --force` recreates the agent (mailbox cleared). `sync -f` resets the code; `rebuild` resets the agent. Destructive levels require an explicit flag.
- **remove --force confirms and cleans up best-effort — no skip-remote flag**: removal is destructive, so — mirroring `rebuild` — a bare `quimby remove <agent>` only warns and removes nothing; `--force` is the confirmation. `--force` then does the **full** teardown, including the remote `rm -rf` for an SSH agent, but **tolerantly**: if the host is unreachable the remote step fails, and rather than abort, quimby removes the local state entry anyway and warns that the remote wasn't cleaned. This replaces the earlier design where `--force` _skipped_ remote cleanup — which left a reachable SSH agent's remote workspace as cruft and offered no "confirm + also clean the remote" path. Folding both into one tolerant `--force` means the common case (reachable) is cleaned and the escape hatch (unreachable) still works, with one flag and no way to accidentally orphan a reachable remote.
- **No artificial simplicity**: This is infrastructure for multi-agent orchestration. Networking, servers, persistent state, SSH transport, and status mirroring are all in scope.
- **The multi-panel dashboard is a three-layer nesting of views over retained sessions**: `quimby run "a b | c d"` arranges agents into split panes, each pane itself a tabbed view. Rather than invent a new sharing primitive, this composes the two tmux facts quimby already relies on — a **session** is the unit of retention, and `link-window` is the unit of sharing — into a strict hierarchy: (1) each **agent** is its own long-lived session (unchanged); (2) a **tab group** (`a b`) is an _ephemeral_ session that `link-window`s those agents' windows, i.e. exactly today's tabbed dashboard; (3) a **pane layout** (`|` `/`) is an _ephemeral_ wrapper session whose single window is split into panes, each pane running a nested `tmux attach` to a tab-group session. So a panel dashboard is an ephemeral view over ephemeral views that bottom out in the retained agent sessions — nothing durable is created, and tearing the whole thing down never touches an agent. The grammar is three layout operators over agent names, precedence tightest→loosest: **space** = tab together in one pane, **`/`** = stack rows, **`|`** = columns; parens override. So `a b | c d` is `(a b) | (c d)` — tabs{a,b} beside tabs{c,d} — and `((a b)/c)|d e` is a left column of tabs{a,b}-over-`c` beside a right pane of tabs{d,e}. A trailing **`:N`** on an agent name or a `)` group close is a **size weight** (see the weight decision below): `(a b):70 / c:30` splits the rows 70:30. Bare `quimby run a b` (no operators) stays the flat tabbed dashboard; the panel path engages only when a layout operator is present, so the feature is purely additive. A **host shell** is a slot like any other, written `host` or the shorthand `$`, and — unlike the flat dashboard's single auto-prepended host tab — it is **repeatable**: `$ | $` is two host panes, `a $` is agent `a` beside a host tab, so you can have as many host terminals as you place.
- **A VS Code extension should render resolved layouts, not reimplement Quimby**: the useful VS Code experience is to open a saved `quimby.yaml` layout as side-by-side named terminals, with agent leaves running `quimby run <agent>` and host/service leaves running their resolved commands. That is a renderer problem, not a new orchestration model. The extension should not parse the layout grammar, merge layered config, resolve presets, infer missing agents, or understand host/service tokens; duplicating those rules would create a second source of truth and subtle drift from `quimby run --layout`. The right seam is a machine-readable `quimby layout <name-or-preset> --json` plan: Quimby owns parsing and validation, the extension owns drawing terminals. VS Code can create named terminals and split one terminal beside another, so it can reproduce the common dashboard cockpit well, but it should treat weights like `:70` as best-effort hints and leave exact weighted pane fidelity to the tmux renderer. tmux remains the retained session substrate; VS Code is another viewport over it.
- **Pane sizes are node weights (`:N`), never percentages, and never touch tabs**: a dashboard node can carry a trailing `:N` — `agentName:N` or a `)` group close `(…):N` — that sets its share of the split it sits in. The `:` sigil is load-bearing: without it a bare number would collide with the `space = tab together` rule and parse as a tab member named `N` (agent names are arbitrary strings, digits included), and it visually binds the size to the **node** rather than burying it among a pane's contents — a size is a property of the panel, not of what's tabbed inside it. Weights are **relative** (sum-and-divide), not percentages: siblings need not total anything, unsized siblings split the remainder equally (`a:70 / b / c` → b and c share the leftover 30), and the model composes cleanly under nesting where a fixed-100 rule would not. A `%` sigil was rejected for implying a must-total-100 that weights don't have. A weight is only meaningful on a node that has siblings under a split axis (`|` or `/`); a weight on a lone pane or on a **tab member** (`(a:2 b:1)` — tabs have no size, only one shows at a time) is an **error**, surfaced rather than silently dropped, so the grammar tells you sizes don't apply to tabs instead of quietly ignoring the `2:1`. The ratio model deliberately mirrors tmux's own percentage-based pane resize, so the wrapper can map weights straight onto pane sizes.
- **The panel wrapper grabs no key binds; the tab layer owns the prefix**: tabbing is the high-frequency action, so `C-b` must reach the _tab_ layer, not be consumed by the pane wrapper. The wrapper session is created with `prefix None` — it never enters prefix mode, so `C-b` passes straight through to the focused pane's nested client (the tab group) where it drives tab switching. Pane management (the rarer op) lives on no-prefix root-table chords (`bind -n M-Left/Right/z`) plus `mouse on` (click-to-focus, drag-to-resize). One prefix to learn, doing the frequent thing. A consequence that falls out cleanly: `C-b d` in a pane detaches that pane's inner client, which (via the teardown below) collapses the whole dashboard — so the detach reflex becomes "cleanly leave the dashboard," agents surviving. This supersedes an earlier two-prefix scheme (`C-Space` for panes) as both simpler and better-aligned with how the dashboard is actually used.
- **Ephemeral views bubble-up teardown; any exit collapses the group, agents survive**: the ephemeral sessions of one dashboard invocation share a name prefix (`qbv-<projectId[:8]>-…`); the agents are a separate namespace. Each pane's command is `tmux attach -t <view>; <teardown>`, so when any child dies or is detached its attach returns and the teardown collapses the entire group — making orphaned views structurally impossible (the failure mode of a top-level-only GC). Two non-obvious requirements, both learned from a prototype and easy to get wrong: the teardown must run **detached on the tmux server** (`run-shell -b`), not inline in the doomed pane, or it kills its own host shell mid-sweep and leaks the siblings; and it must kill the wrapper session **last**. Because agents are retained, an aggressive "any exit collapses the viewport" default is the right one — re-running the dashboard is instant — rather than a gentler per-pane close. A separate, non-obvious nesting requirement: a pane running `tmux attach` inherits `$TMUX` and tmux refuses to nest, so the pane silently dies unless the command clears it (`sh -c 'TMUX= tmux … attach …'`).
